Tang Nano 20K HDMI Implementation Detail

This document explains how the Commodore 64 VIC-II video signal is transformed and transmitted over HDMI on the Tang Nano 20K FPGA board.

1. High-Level Architecture

The video pipeline consists of four main stages:

  1. Source Generation: The VIC-II core generates standard definition (15kHz) digital video.
  2. Clock Generation: The FPGA PLL generates the specific pixel clocks needed for HDMI.
  3. Scan Doubling: Converts the 15kHz signal to 31kHz (VGA-compatible) effectively doubling the vertical resolution.
  4. HDMI Encoding & Serialization: Encodes the video data into TMDS packets and serializes them for high-speed transmission.

2. Detailed Verification

Step 1: The Source (VIC-II)

  • File: src/video_vicII_656x.vhd (implied)
  • Function: Works exactly like the original chip. It outputs horizontal and vertical sync pulses (hsync, vsync) and a 4-bit color index.
  • Timing: It runs at ~1MHz (system clock) but generates video timing compatible with PAL (50Hz) or NTSC (60Hz) TV standards.

Step 2: Clock Generation

  • File: src/tang_nano_20k_c64_top.vhd
  • Mechanism: A Gowin rPLL primitive is used to generate two synchronized clocks from the 27MHz board oscillator.
    • clk_pixel (~32 MHz): The pixel clock for the HDMI video mode.
    • clk_pixel_x5 (~160 MHz): 5x the pixel clock, used for shifting out bits serially.
  • Switching: The PLL is dynamically reconfigured (via IDSEL/FBDSEL ports) when switching between PAL and NTSC modes to ensure exact framerates.

Step 3: The Video Wrapper & Scandoubler

  • File: src/misc/video.v and src/misc/scandoubler.v
  • Problem: Modern HDMI monitors/TVs typically cannot sync to 15kHz (standard TV resolution). They expect at least 31kHz (VGA resolution).
  • Solution: The scandoubler module buffers incoming lines.
    • Input: Writes incoming pixels into a line buffer (sd_buffer) at the slow ~15kHz rate.
    • Output: Reads pixels out of the buffer at twice the speed (using the faster clk_sys or derived clock in the module).
    • Result: Each line is effectively displayed, then the next line is displayed, doubling the horizontal frequency.

Step 4: HDMI Encoding

  • File: src/hdmi/hdmi.sv
  • TMDS Encoding: Converts 8-bit color data (Red, Green, Blue) into 10-bit "TMDS characters" using 8b/10b encoding. This minimizes transitions (to reduce EMI) and ensures DC balance.
  • Data Islands: In addition to video, it can send "Data Islands" during the blanking intervals (when no picture is sent). This is used to transmit:
    • Audio: Digital audio samples (from SID) are packetized and sent here.
    • InfoFrames: Metadata about the video format.
  • Synchronization: The video_analyzer.v module monitors the input sync signals and resets the HDMI encoder counters (vreset) to keep the HDMI output locked to the C64 video frames.

Step 5: Serialization

  • File: src/hdmi/serializer.sv
  • Function: Takes the 10-bit parallel data from the encoder and shifts it out bit-by-bit at very high speed (DDR).
  • Primitive: On the Tang Nano 20K (Gowin FPGA), it uses the OSER10 primitive.
    • Inputs: 10 bits of data, PCLK (slow), FCLK (fast).
    • Output: 1 serial bit stream per channel (3 color channels + 1 clock channel).
  • Physical: The TLVDS_OBUF (in video.v) drives the physical differential pairs connected to the HDMI port.

3. Summary of Signal Flow

  1. VIC-II says "Color 5, Pixel X" (Symbolic).
  2. Scandoubler stores it, then spits it out twice as fast.
  3. OSD checks "Am I drawing a menu here?". If yes, replaces pixel.
  4. HDMI Encoder takes the pixel (RGB 8-8-8) and turns it into three 10-bit numbers.
  5. Serializer shoots those 10-bit numbers out the HDMI port at 160 million bits per second.

4. Technical Specifications

Clock Frequencies

The Tang Nano 20K uses a dynamic PLL that switches between PAL and NTSC modes:

Mode Input IDIV FBDIV ODIV clk_pixel_x10 clk_pixel_x5 clk_pixel (clk32) clk64
PAL 27 MHz 2 30 2 315 MHz 157.5 MHz 31.5 MHz 63 MHz
NTSC 27 MHz 4 4 2 ~329.4 MHz ~164.7 MHz ~32.94 MHz ~65.88 MHz

Formula: Output = Input × (FBDIV / IDIV) / ODIV

The FSM in the top-level entity monitors ntscMode and reconfigures the PLL dynamically:

  • When switching to PAL: IDSEL <= "111101" (2), FBDSEL <= "011101" (30)
  • When switching to NTSC: IDSEL <= "111011" (4), FBDSEL <= "000011" (4)

HDMI Video Modes

The HDMI encoder supports three timing modes:

PAL Mode (50 Hz)

  • Frame: 1008 × 624 pixels
  • Active: 800 × 576 pixels
  • HSync: 24 pixels start, 72 pixels width
  • VSync: 5 lines start, 5 lines height
  • CEA: Mode 17

NTSC Mode (60 Hz)

  • Frame: 1040 × 526 pixels
  • Active: 800 × 480 pixels
  • HSync: 16 pixels start, 62 pixels width
  • VSync: 9 lines start, 6 lines height
  • CEA: Mode 2

Wide Screen Variants

Both modes have "wide" variants that add black borders (via start_x offset) to center the image on 16:9 displays while maintaining 4:3 aspect ratio.

TMDS Encoding Details

Each pixel goes through 8b/10b encoding:

  1. Count transitions: Count number of 1s in the 8-bit input (N1D)
  2. Choose encoding method:
    • If N1D > 4 or (N1D == 4 && data[0] == 0): Use XNOR encoding
    • Otherwise: Use XOR encoding
  3. DC balancing: Track running disparity (acc) and invert bits if needed to maintain DC balance
  4. Output: 10-bit character where:
    • Bit [8]: Encoding method flag
    • Bit [9]: Inversion flag
    • Bits [7:0]: Encoded data

Control Symbols (during blanking):

  • hsync=0, vsync=0: 10'b1101010100
  • hsync=1, vsync=0: 10'b0010101011
  • hsync=0, vsync=1: 10'b0101010100
  • hsync=1, vsync=1: 10'b1010101011

Serialization on Gowin FPGA

The serializer uses Gowin's OSER10 primitives:

OSER10 gwSer0(
    .Q(tmds[0]),           // Serial output
    .D0-D9(...),           // 10 parallel bits
    .PCLK(clk_pixel),      // ~32 MHz pixel clock
    .FCLK(clk_pixel_x5),   // ~160 MHz fast clock
    .RESET(reset)
);

Operation:

  • Loads 10 bits on each PCLK edge
  • Shifts out 2 bits per FCLK edge (DDR - both edges)
  • Effective serial rate: 160 MHz × 2 = 320 Mbps per channel

The differential outputs are driven by TLVDS_OBUF primitives:

TLVDS_OBUF tmds_bufds [3:0](
    .I({tmds_clock, tmds}),
    .O({tmds_clk_p, tmds_d_p}),
    .OB({tmds_clk_n, tmds_d_n})
);

Audio Integration

Audio samples from the SID chip are embedded in HDMI Data Islands:

  • Sample rate: 48 kHz (generated by dividing clk_pixel by audio_div)
  • Bit depth: 16-bit stereo
  • Format: L-PCM (Linear Pulse Code Modulation)
  • Packets: Audio Sample Packets and Audio Clock Regeneration packets

The packet_picker.sv module schedules when to send audio packets during horizontal blanking intervals.
