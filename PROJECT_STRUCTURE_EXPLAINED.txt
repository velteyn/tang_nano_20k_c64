FPGA C64 Project Structure Analysis

1. The "Onion" Structure
The project is organized in layers, similar to a real computer system:
* The Core (Common to all): Contains the Commodore 64 chips (6510 CPU, VIC-II video chip, SID audio chip, etc.). These are described in HDL (VHDL/Verilog) and are pure "logical concepts". They are unaware of being on a Tang Nano or a Tang Primer.
* The Adapter (Top Level): This is the board-specific file acting as a "bridge". It takes the Core's logical wires and connects them to the physical pins of your specific board.
* The Project (.gprj): This is the "shopping list" telling the compiler which files to combine to create the final bitstream for *that specific* board.

2. File Analysis

Common Files (The Core)
Found in the "src/" directory, these are shared across all projects. Modifying one reflects on all boards because they all point to the same physical file.
* Chips: cpu_6510.vhd, video_vicII_656x.vhd, mos6526.v (CIA), sid/ (Audio).
* Logic: fpga64_buslogic.vhd (bus handling), c1541/ (floppy drive emulation).

Specific Files (The Differences)
Each board has dedicated files handling hardware differences.
* Gowin Project (.gprj): Comparing "tang_nano_20k_c64.gprj" vs "tang_primer_25k_c64.gprj", you'll see they include the same 90% of common files but select different files for memory and top level.
* Top Level Entity (*_top.vhd):
    * Tang Nano 20k: Uses "src/tang_nano_20k_c64_top.vhd".
    * Tang Primer 25k: Uses "src/tang_nano_20k_c64_top_25k.vhd".
* Physical Constraints (.cst): Maps logical signals to the FPGA chip's physical pins.

3. Focus: How Video Works (VIC-II -> HDMI)
Video handling is a perfect example of how the project manages complexity by sharing code.

Step 1: The Logical Core (VIC-II)
 The file "src/video_vicII_656x.vhd" is the "brain".
* It generates pure digital video signals: hSync, vSync, and a 4-bit color index.
* It knows nothing about HDMI. It believes it's talking to an old CRT.
* It is shared code: identical for all boards.

Step 2: The Video Adapter (video.v)
The file "src/misc/video.v" acts as a universal translator.
* Scandoubler: The VIC-II outputs at 15kHz (too slow for modern monitors). The "scandoubler" module doubles the lines to reach 31kHz (VGA/HDMI compatible).
* HDMI/TMDS: The "src/hdmi/hdmi.sv" module takes the doubled pixels and converts them into the high-speed serial protocol (TMDS) that travels over the HDMI cable.
* Physical Driver: Uses a primitive called "TLVDS_OBUF" to drive the FPGA's differential pins.

Step 3: The Clock Magic (The Real Difference)
This is the trick. HDMI requires precise clocks (e.g., 160MHz for serial pixels). Each board has a different oscillator and a different FPGA chip generating clocks differently.
* Tang Nano 20k: Its top-level file uses an "rPLL" component (specific to that chip) to generate the "clk_pixel_x5" signal.
* Tang Primer 25k: Its top-level file uses "Gowin_PLL_pal"/"ntsc" components (specific to the GW5A series) to generate the exact same "clk_pixel_x5" signal.

Result: Both top-level files pass the exact same "clk_pixel_x5" signal to the "video.v" module, even though it was generated by different hardware. The (shared) video module uses this clock to drive HDMI without knowing its origin.

Summary
* If you change VIC-II logic: You change "video_vicII_656x.vhd", affecting all boards.
* If you change how the clock is generated: You modify only your specific board's "*_top.vhd" file.
* HDMI Video: Is handled by a common module (video.v + hdmi/) that only expects to receive the correct clock. It is the specific Top Level file's job to provide that clock.
